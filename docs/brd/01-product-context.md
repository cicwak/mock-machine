# Product Context: Mock Machine

## Назначение

Mock Machine - сервис для настройки HTTP-моков, которые возвращают заранее заданные данные на выбранные пути. Основная цель продукта - ускорить разработку и тестирование клиентских и интеграционных сценариев, когда реальный backend недоступен, нестабилен, не готов или должен отдавать редкие ошибки.

## Пользователи

- Backend-разработчики, которым нужно быстро имитировать смежные сервисы.
- Frontend-разработчики, которым нужны стабильные ответы для экранов и состояний UI.
- QA-инженеры, которым нужно воспроизводить успешные, ошибочные и timeout-сценарии.
- Интеграционные команды, которым нужно видеть, какие запросы приложение уже отправляет, но какие моки еще не настроены.
- LLM-агенты и автоматизации, которым нужно программно создавать и обновлять моки.

## Основные пользовательские сценарии

1. Пользователь открывает админ-панель без авторизации.
2. Пользователь видит дерево мок-маршрутов, например `/api` -> `/v1` -> `/account` -> `/{account_id}`.
3. Пользователь создает маршрут, выбирает HTTP-метод, задает path pattern, status code, headers и body.
4. Пользователь добавляет несколько сценариев ответа для одной ручки: success, error, timeout или custom.
5. Пользователь быстро переключает активный сценарий и сразу получает новое поведение mock service.
6. Приложение отправляет запрос на неизвестный путь.
7. Mock Machine сохраняет этот запрос как неразобранный и показывает его в панели.
8. Пользователь создает полноценный мок-маршрут из неразобранного запроса с предзаполненными данными.
9. Автоматизация или LLM-агент создает и обновляет моки через административное API или MCP server.

## Ценность

- Меньше ручного создания моков с нуля.
- Быстрое переключение между успехом, ошибками и timeout-состояниями.
- Единый входной порт для frontend-панели и мок-сервиса.
- Возможность автоматизировать настройку моков без ручной работы в панели.
- Версионируемое описание требований и архитектурных решений прямо в репозитории.

## Границы первой версии

- Админ-панель доступна без авторизации.
- Unknown-запросы не проксируются в реальный upstream.
- Сложный scripting/rules engine не входит в первую версию.
- MinIO поднимается в инфраструктуре сразу, но полноценное мокирование файлов может быть реализовано отдельным этапом.
