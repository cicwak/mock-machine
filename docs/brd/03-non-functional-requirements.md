# Non-Functional Requirements

## Runtime and Deployment

Система должна запускаться через Docker Compose.

Состав окружения:

- nginx;
- frontend;
- backend mock service;
- PostgreSQL;
- Redis;
- MinIO.

nginx должен быть единственной внешней точкой входа:

- `/mockadmin` проксируется во frontend;
- `/mockadminapi` проксируется в backend administrative API;
- остальные пути проксируются в backend mock service.

## Persistence

PostgreSQL является основным хранилищем:

- мок-маршруты;
- сценарии;
- настройки активных сценариев;
- unknown requests;
- история первого и последнего появления unknown request.

Redis используется для:

- кеша активных маршрутов;
- быстрого переключения активных сценариев;
- снижения количества обращений к PostgreSQL в runtime path.

MinIO используется как S3-compatible object storage для будущего мокирования файлов и бинарных ответов.

## Performance

Mock service должен обрабатывать runtime-запросы без обязательного обращения к PostgreSQL на каждый запрос, если маршрут уже есть в кеше.

Ожидаемые требования первой версии:

- быстрое чтение активных маршрутов;
- переключение сценария без перезапуска сервиса;
- группировка unknown requests без создания неограниченного числа дублей;
- корректная работа при повторяющихся запросах на один неизвестный path.

## Reliability

Сервис должен сохранять unknown requests даже если для них нет маршрута.

Если Redis недоступен, backend должен иметь возможность читать конфигурацию из PostgreSQL, пусть и с более высокой задержкой.

Если MinIO недоступен, обычные JSON/text моки должны продолжать работать.

## Security

В первой версии авторизация админ-панели не требуется.

При этом необходимо:

- явно отделить административное API от публичных мок-ручек;
- не использовать `/mockadmin` как путь для мок-ответов;
- не использовать `/mockadminapi` как путь для мок-ответов;
- учитывать, что сохраненные headers/body unknown-запросов могут содержать чувствительные данные;
- предусмотреть возможность последующего добавления авторизации без полной переработки API.

## Observability

Backend должен иметь health check endpoint.

Логи должны позволять понять:

- какой маршрут был найден;
- какой сценарий был выбран;
- когда запрос попал в unknown requests;
- были ли ошибки записи в PostgreSQL, Redis или MinIO.

## Automation Interfaces

Все операции, доступные в админ-панели, должны иметь эквивалент в `/mockadminapi`.

MCP server должен быть отдельным интерфейсом для LLM-клиентов и должен использовать те же доменные операции, что и административное API. Это снижает риск расхождения поведения между web-панелью, API-автоматизацией и LLM-автоматизацией.

## Compatibility

Первая версия ориентирована на локальную разработку и docker-compose окружение.

Требования не предполагают production-grade multi-tenant модель, сложный RBAC или внешний identity provider.
