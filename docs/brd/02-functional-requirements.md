# Functional Requirements

## Админ-панель

Панель должна быть доступна по префиксу `/mockadmin` и работать без авторизации в первой версии.

Панель должна позволять:

- просматривать список мок-маршрутов;
- создавать, редактировать и удалять мок-маршруты;
- отображать маршруты деревом сегментов пути;
- искать маршруты по path, HTTP-методу, status code, названию и тегам;
- просматривать сценарии ответа конкретного маршрута;
- переключать активный сценарий маршрута;
- видеть отдельный список неразобранных запросов;
- создавать мок-маршрут из неразобранного запроса.

Дерево путей должно группировать маршруты по сегментам:

- `/api` как корневой узел;
- `/api/v1` и `/api/v2` как дочерние узлы;
- `/api/v1/account/{account_id}` как конечный или промежуточный узел.

## Мок-маршруты

Маршрут должен определяться минимум следующими полями:

- HTTP method;
- path pattern;
- название;
- optional tags;
- список сценариев ответа;
- активный сценарий по умолчанию.

Path pattern должен поддерживать:

- статические сегменты, например `/api/v1/accounts`;
- динамические сегменты, например `/api/v1/account/{account_id}`;
- извлечение значений динамических сегментов в path params.

Приоритет поиска маршрута:

1. Точное совпадение path.
2. Параметризованное совпадение path.
3. Unknown request capture, если маршрут не найден.

## Сценарии ответа

Один маршрут должен поддерживать несколько сценариев ответа. В первой версии обязательны следующие типы:

- `success`;
- `error`;
- `timeout`;
- `custom`.

Сценарий должен позволять настраивать:

- status code;
- response headers;
- response body;
- delay/timeout behavior;
- простые условия выбора сценария.

Для быстрых ошибок должны быть доступны шаблоны:

- 400 Bad Request;
- 401 Unauthorized;
- 403 Forbidden;
- 404 Not Found;
- 409 Conflict;
- 422 Unprocessable Entity;
- 429 Too Many Requests;
- 500 Internal Server Error;
- 503 Service Unavailable.

Активный сценарий должен переключаться без пересборки сервиса и без перезапуска контейнеров.

## Динамические параметры и шаблоны

Сервис должен уметь подставлять значения из запроса в ответ.

Минимально поддерживаемые источники:

- path params, например `{account_id}`;
- query params;
- request headers;
- JSON body fields, если тело запроса является JSON.

Пример:

- маршрут: `/api/v1/account/{account_id}`;
- запрос: `GET /api/v1/account/123`;
- response body может вернуть `{"id":"123"}`.

В первой версии достаточно простого шаблонизатора с явными placeholders. Сложные вычисления, циклы и пользовательский код не входят в базовый объем.

## Unknown Requests

Если входящий запрос не совпал ни с одним мок-маршрутом, backend должен:

1. Сохранить запрос как неразобранный.
2. Сгруппировать повторные запросы с тем же method и path.
3. Увеличить счетчик повторов.
4. Обновить timestamp последнего запроса.
5. Сохранить последний пример query, headers и body.
6. Вернуть клиенту `404 route is not configured`.

Unknown request должен хранить:

- HTTP method;
- path;
- query;
- headers;
- body;
- first seen timestamp;
- last seen timestamp;
- count;
- статус обработки: new, ignored или converted.

Админ-панель должна позволять создать мок-маршрут из unknown request. При создании должны быть предзаполнены method, path, query, headers и body последнего примера.

## Backend Mock Service

Backend должен принимать все запросы, кроме административных API и frontend-префикса `/mockadmin`.

Поведение:

- если маршрут найден, вернуть ответ выбранного сценария;
- если сценарий содержит delay, задержать ответ на заданное время;
- если сценарий является timeout, имитировать отсутствие ответа до заданного лимита;
- если маршрут не найден, создать или обновить unknown request;
- административные endpoints не должны конфликтовать с публичными мок-ручками.

## Административное API

Административное API должно быть отделено от публичных мок-ручек обязательным префиксом `/mockadminapi`.

Весь функционал админ-панели должен быть доступен через административное API. Панель является web-клиентом поверх API, а не отдельным источником бизнес-логики. Это нужно, чтобы маршруты, сценарии, unknown requests и настройки можно было создавать, читать, обновлять и удалять автоматизированно.

Минимальные группы endpoints:

- управление маршрутами;
- управление сценариями;
- переключение активного сценария;
- просмотр unknown requests;
- создание маршрута из unknown request;
- поиск и получение дерева маршрутов;
- управление шаблонными ошибками и timeout-сценариями;
- health check.

Административное API должно быть пригодно для внешней автоматизации:

- стабильные JSON request/response контракты;
- явные ошибки валидации;
- идемпотентные операции там, где это применимо;
- возможность создать маршрут и сценарии без открытия web-панели.

## MCP Server

Mock Machine должен поддерживать MCP server, чтобы LLM-клиенты могли программно работать с мок-сервисом.

MCP server должен предоставлять инструменты для ключевых операций панели:

- получить список маршрутов;
- получить дерево маршрутов;
- найти маршрут по path/method;
- создать маршрут;
- обновить маршрут;
- удалить маршрут;
- создать или обновить сценарий ответа;
- переключить активный сценарий;
- получить список unknown requests;
- создать маршрут из unknown request;
- получить health/status сервиса.

MCP server должен использовать те же backend-возможности, что и `/mockadminapi`, чтобы поведение панели, API и LLM-автоматизации не расходилось.
