# ADR 0003: Unknown Request Capture

## Status

Accepted

## Context

Одна из ключевых фич Mock Machine - не заставлять пользователя заранее руками описывать все маршруты. Если приложение уже отправляет запросы, которые не описаны в моках, сервис должен показать эти запросы в админ-панели.

## Decision

Если backend mock service получает запрос, для которого нет настроенного маршрута, он должен:

- сохранить или обновить unknown request;
- сгруппировать повторы по method и path;
- сохранить последний пример query, headers и body;
- увеличить счетчик повторов;
- вернуть клиенту `404 route is not configured`;
- показать unknown request в отдельном разделе админ-панели.

Пользователь должен иметь возможность создать мок-маршрут из unknown request. После создания unknown request получает статус `converted`.

## Rationale

Это превращает ненастроенные запросы в backlog для настройки моков и уменьшает ручной ввод. Пользователь видит реальные path, method, headers и body, а не восстанавливает их по логам или коду клиента.

Возврат явного `404 route is not configured` делает поведение предсказуемым: запрос не проксируется во внешний upstream и не маскируется дефолтным успешным ответом.

## Consequences

- Unknown requests должны храниться в PostgreSQL.
- Повторные unknown-запросы не должны создавать бесконечные дубли.
- Для privacy/security нужно учитывать, что headers и body могут содержать чувствительные данные.
- В будущих версиях можно добавить ignore rules, TTL, masking headers и upstream fallback, но они не входят в первую версию.
